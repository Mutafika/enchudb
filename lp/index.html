Six primitives. That's the whole engine.

Everything else (SQL frontend, RAG, transport, sync) builds on these. Read the engine source in an afternoon.

core

紐 (himo) — attribute as index

An entity has himos. A himo is a typed attribute (Symbol / Value / Text) that's indexed by construction. There is no "should I add an index" question — every himo is one.

v27

BucketCylinder

O(1) insert and remove on unsorted buckets. Replaces v26's sorted lists. The data structure that makes tie 85ns and pull_raw 105ns regardless of cardinality.

v26

PairTable

2D dense cells indexed by (himo_a × himo_b). 2-condition AND becomes a single cell read instead of intersect-of-sets. This is the 421× lever.

v28+

WAL with ring buffer

Crash consistency without a separate log file growing forever. Background fsync, ring-buffer reuse. Sync mode 148µs, Async mode 100ns. Crash-safe by default.

v31+

Ravn — graph traversal DSL

Multi-step follow → filter → reverse_follow over himos as edges. "Decisions made by Alice in Next.js sessions" reads like a path expression, runs like a join.

add-on

RAG store on top

Brute-force cosine + meta filter via PairTable. No ANN, no recall loss. Filter narrows N before vectors do — that's why 0.49ms holds at 100k chunks.

Three lines to a query.

Add to Cargo.toml, define your himos, start tieing. The engine is one mmap file plus a WAL — no daemon, no migration.

01
Add the depenchudb = { git = "https://github.com/mutafika/enchudb" }
02
Open or createEngine::create(path) for a fresh DB, Engine::open for existing.
03
Define himos, tie, queryTie attributes to entities, call rebuild(), query with AND clauses.

main.rs 
use enchudb::{Engine, HimoType, Ravn};

let mut db = Engine::create("/tmp/my.db")?;
db.define_himo("age", HimoType::Value, 100);
db.define_himo("city", HimoType::Text, 10_000);

// tie attributes to an entity
let alice = db.entity();
db.tie(alice, "age", 30);
db.tie_text(alice, "city", "東京");

db.rebuild();

// 187ns multi-condition AND
let hits = db.query(&[("age", 30), ("city", vocab("東京"))]);

// graph traversal: "decisions made by Alice"
let ravn = Ravn::new(db.clone());
let sessions  = ravn.reverse_follow(&hits, "speaker");
let decisions = ravn.extract_text(&sessions, "decision");

Stop indexing. Just tie it.

If filtering by attributes is your hot path, the indexing question is wasting your time. EnchuDB makes attributes the index — no plans, no ANALYZE, no recall loss.
